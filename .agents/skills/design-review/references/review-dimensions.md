# 审查维度检查清单

四个审查维度的详细检查项。审查时不必逐条检查，根据方案类型选择相关项即可。

## 完整性（Completeness）

方案是否覆盖了所有应该覆盖的场景。

- 需求覆盖：所有功能需求是否都有对应实现
- 边界条件：空值、零值、极大值、空列表等边界是否处理
- 异常路径：网络超时、服务不可用、数据格式错误等异常是否有处理逻辑
- 接口完整：对外暴露的 API 是否都有入参校验和错误响应定义
- 状态流转：如果涉及状态机，所有状态转换路径是否完整
- 回滚机制：操作失败时是否有回滚或补偿策略
- 文档同步：代码变更是否需要同步更新文档或配置

## 可行性（Feasibility）

方案在技术上是否可以落地。

- 依赖可用：所需的外部服务、库、API 是否可用且版本兼容
- 性能预期：在预期数据量和并发量下是否能满足性能要求
- 资源消耗：内存、CPU、网络带宽是否在合理范围内
- 部署约束：是否兼容现有的部署环境和 CI/CD 流程
- 数据迁移：如果涉及数据结构变更，迁移方案是否可行
- 时间复杂度：核心算法的时间复杂度是否可接受
- 并发安全：多线程/多进程场景下是否有竞态条件

## 安全性（Security）

方案是否存在安全隐患。

- 注入攻击：SQL 注入、命令注入、XSS、SSRF 等
- 认证授权：接口是否有适当的身份验证和权限检查
- 敏感信息：密钥、Token、密码是否安全存储，日志中是否脱敏
- 输入校验：外部输入是否经过充分验证和清理
- 依赖安全：引入的第三方库是否有已知漏洞
- 数据传输：敏感数据传输是否使用加密通道
- 错误暴露：错误信息是否泄露内部实现细节

## 可维护性（Maintainability）

方案是否便于后续维护和演进。

- 代码结构：是否遵循项目约定的分层架构
- 职责清晰：每个模块/类/函数是否职责单一
- 命名规范：变量、函数、类的命名是否清晰表达意图
- 类型安全：公共函数是否有完整的类型注解
- 可测试性：核心逻辑是否可以独立测试，依赖是否可 mock
- 配置管理：是否通过统一配置管理，而非硬编码或直接读环境变量
- 日志规范：关键操作是否有适当的日志记录
- 扩展性：如果需求变化，修改范围是否可控
