# Design Review（方案审查）

## 概述

跨工具通用的方案审查工作流。上下文隔离通过**跨会话/跨工具**实现：方案由一个 AI 会话生成，审查由另一个 AI 会话（可以是不同工具、不同模型）独立执行，天然隔离作者上下文，确保审查客观性。

本 skill 不依赖任何平台专属工具，只需：读文件、执行 shell 命令、文本对话交互。

流程循环：获取方案 → 独立审查 → 展示 findings → 用户指定修复项 → 执行修复 → 可选继续审查

## 工作流

### Step 1: 确定审查范围

获取待审查内容：

**优先：git diff**（适合审查代码变更）
```bash
git diff --stat
git diff
```

**备选：读取指定文件/目录**（适合审查设计文档、新建文件）

如果 git diff 为空且用户未指定文件，直接询问用户：

> "当前没有未提交的变更，请告诉我审查范围：
> 1. 暂存区变更（git diff --cached）
> 2. 对比主分支（git diff main...HEAD）
> 3. 指定文件路径"

等待用户回复后继续。

### Step 2: 独立审查

**重要**：执行本步骤时，将自身切换为独立审查者视角——忽略本次对话中关于方案生成的上下文，仅基于当前读取到的方案内容进行审查，如同第一次看到这份方案。这模拟了跨会话/跨工具审查的上下文隔离效果。

参照 `references/review-dimensions.md` 的检查清单，从四个维度审查：

- **完整性**：需求覆盖、边界条件、异常路径、接口完整性
- **可行性**：技术可落地性、依赖可用性、性能可接受性
- **安全性**：注入风险、认证授权、敏感信息处理
- **可维护性**：代码结构、职责清晰度、规范遵循、可扩展性

**每个 finding 的格式**：

```
[F{序号}] [{维度}] [{严重程度}] 标题
- 问题描述：具体说明问题所在
- 建议修复：如何修复此问题
- 涉及位置：文件路径:行号 或 章节名（如适用）
```

严重程度定义：
- **P0**（Critical）：必须修复（安全漏洞、数据丢失风险、正确性 bug）
- **P1**（High）：建议修复（重大设计缺陷、逻辑错误、性能回退）
- **P2**（Medium）：本轮或后续修复（可维护性问题、轻微规范违反）
- **P3**（Low）：可选改进（风格、命名、轻微优化）

### Step 3: 展示 findings 并询问修复项

将所有 findings 按严重程度排序（P0 → P1 → P2 → P3）完整展示。

展示后，询问用户：

> "共发现 N 个问题（P0: _, P1: _, P2: _, P3: _）。
> 请回复需要修复的 finding 编号，例如：1,3,5
> 或回复：all（全部修复）、p1（仅修复 P0+P1）、skip（跳过修复）"

等待用户回复。

### Step 4: 执行修复

根据用户指定的编号，逐个修复选中的 findings：

1. 读取涉及的文件
2. 理解问题上下文
3. 实施修复（如项目有 Agent 指令文件则优先参照其规范）
4. 简要说明每项修复内容

修复完成后，展示变更摘要：修复了哪些问题、改动了哪些文件。

### Step 5: 询问是否继续审查

询问用户：

> "修复已完成。是否对修复后的方案再次审查？（yes/no）"

- 用户回复 **yes**：回到 Step 1，开始新一轮审查（基于最新文件状态）
- 用户回复 **no**：审查流程结束

## 跨工具使用说明

本 skill 可在以下环境中直接运行，无需任何修改：

| 环境 | 运行方式 |
|------|---------|
| Claude Code | 通过 `/design-review` 触发 |
| Codex CLI | 在 Codex 会话中粘贴 SKILL.md 内容作为指令 |
| Cursor | 在 Agent 模式下引用本 skill |
| Trae IDE | 在 AI 对话中引用本 skill |

**推荐工作流**（最大化上下文隔离）：

```
会话 A（任意工具）：生成方案 → 保存到文件
         ↓
会话 B（不同工具或新会话）：执行 design-review skill 审查
```

不同模型、不同会话之间的隔离效果优于同会话内的"视角切换"。
