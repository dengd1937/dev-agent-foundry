# Self-Review Skill

> 在需要时进行任务驱动的本地自审，验证变更是否真正完成需求、未破坏架构约束，并具备足够的本地证据支持下一步 review 或提交。

## 使用时机

**当用户明确要求额外本地检查时，运行此 skill。**

### 判断逻辑

在运行 self-review 之前，先检查变更内容：

```bash
# 查看工作区与暂存区变更
git diff --name-only
git diff --cached --name-only
```

| 变更类型 | 操作 |
|---------|------|
| 代码、测试、脚本、CI、容器、配置、依赖、接口契约、数据库/工作流定义 | 运行 self-review |
| 行为性文档改动：会改变操作方式、发布方式、流程、排障方式、验收标准或 agent 行为 | 运行 self-review |
| 纯说明性文档改动，且不影响行为、流程或验收 | 可跳过 self-review |

**一旦选择运行本 skill，自审不通过时不应继续提交。**

## 核心原则

- `self-review` 不是一次性 checklist，而是提交前的第一层质量闭环。
- 目标不是"跑完几个命令"，而是回答：**这次改动是否真的完成任务，并且有足够证据证明没有明显回归。**
- 优先依赖可执行证据：测试、脚本、构建、日志、接口返回、页面行为、结构化检查。
- 如果发现问题，应该先修复，再重新运行相关验证；不能带着已知失败继续提交。
- 若用户请求的是交付动作，必须把仓库要求的交付流程一并纳入检查，而不是只验证代码本身。
- `self-review` 通过后，复杂改动仍应继续进入更严格的 review，例如 `code-review-expert` 或人工 review。

## 自审流程

按顺序执行以下五步：

### 1. 明确任务与验收标准

先回答以下问题，并在最终报告中给出简要结论：

- 这次改动要解决什么问题，或者交付什么能力
- 影响哪些模块、接口、脚本、配置或文档
- 成功标准是什么
- 哪些风险最值得优先验证

如果是 Bug 修复，必须先确认以下至少一项：

- 已复现原问题
- 已拿到稳定的失败样本、错误日志或最小复现场景
- 若无法复现，已明确说明缺少什么证据，且不能把"猜测修复"当作通过

### 2. 制定最小充分验证计划

根据任务类型选择验证动作，而不是固定只跑一套命令。

| 任务类型 | 最低验证要求 |
|---------|-------------|
| 代码改动 | 格式检查、lint 检查、相关测试 |
| Bug 修复 | 先复现失败，再验证修复；至少保留一条"修复前失败 / 修复后通过"的证据 |
| 接口或数据模型改动 | 相关单测/集成测试、示例请求响应、字段兼容性检查 |
| 配置/部署/CI/脚本改动 | 对应脚本、构建、启动、健康检查或 dry-run |
| 工作流/外部系统集成改动 | 相关端到端路径验证、关键日志、关键返回值或模拟调用 |
| 行为性文档改动 | 校验文档与当前实现、命令、路径、配置名、流程步骤是否一致 |
| 纯说明性文档改动 | 可跳过，或仅做最小一致性检查 |

如果现有仓库缺少必要测试：

- 先补最小必要测试或验证脚本
- 若短期无法补齐，必须在报告中明确记录风险与原因
- 对高风险改动，**没有验证证据就不应提交**

### 3. 执行机械化检查

以下检查优先使用可执行命令完成，避免主观判断。

#### 代码质量与基础门禁

运行项目约定的格式化和 lint 工具。常见示例：

```bash
# Python 项目示例
ruff format --check .
ruff check .

# JavaScript/TypeScript 项目示例
npx eslint .
npx prettier --check .

# Go 项目示例
go fmt ./...
go vet ./...
```

- 必查项：
  - 格式检查通过
  - Lint 检查通过
  - 变更相关测试通过
  - 文件大小约束未被破坏（如项目有规定）
  - 公共函数类型注解满足仓库规范（如项目有规定）

#### 架构与仓库约束

至少核对以下内容：

- 分层依赖是否被破坏
- 配置访问方式是否符合项目约定
- 无新增循环依赖
- docstring / 注释 / 配置文档已与代码保持一致
- 未引入调试用的 `print` / `console.log` 等

建议优先使用搜索工具做快速筛查。

#### 安全与可观测性

至少核对以下内容：

- 未硬编码 API Key、密码、Token 等敏感信息
- 日志未泄露敏感字段
- 新增错误路径带有足够定位信息
- 外部输入有基本校验

### 4. 执行任务相关验证

根据第 2 步的验证计划执行真实验证。常见方式包括：

- 运行针对性的测试用例，而不是无差别只跑全量
- 调用本地接口或脚本，核对关键输出
- 对 bug 修复保留失败前后的对比证据
- 检查关键日志、状态变更、错误处理和回退路径
- 必要时补最小测试，避免仅靠肉眼判断

这里的通过标准是：

- 改动目标被直接验证
- 主要风险点至少各有一条证据
- 没有未解释的失败、跳过或偶发波动

### 5. 进行差异审查并决定是否允许提交

在命令通过后，还要对 diff 做一次面向风险的人工审查：

- 是否存在"任务没做完但测试恰好没覆盖"的情况
- 是否为了让测试通过而削弱了原有行为
- 是否修改了注释、文档、配置却没同步实现
- 是否引入了不必要的复杂度、重复逻辑或临时代码
- 若任务是交付类请求，是否已经满足仓库要求的流程闭环
- 是否需要额外 review

满足以下任一条件时，建议继续运行 `code-review-expert` 或请求人工 review：

- 跨模块重构
- 外部系统集成改动
- 权限、认证、告警、配置、数据模型改动
- 测试覆盖不足但必须先推进
- 自审过程中出现过难以解释的失败

## 输出格式

检查完成后，在终端输出结构化报告，必须覆盖以下字段：

```
Self-Review 报告

任务目标：
- <本次改动要完成什么>

变更范围：
- <关键文件/模块>

验证计划：
- <本次选择了哪些检查，为什么足够>

执行结果：
- [PASS/FAIL] 格式与 lint
- [PASS/FAIL] 相关测试
- [PASS/FAIL] 任务相关验证
- [PASS/FAIL] 架构/安全/文档一致性检查

关键证据：
- <例如：某测试通过、某接口返回、某日志片段、某失败已复现后消失>

遗留风险：
- <若无则写"无">

结论：
- PASS: 允许提交
或
- FAIL: 不允许提交

后续动作：
- <是否需要 code-review-expert / 人工 review / 补测试>
- <若为交付类任务，是否还需要 push / 创建 PR>
```

## 失败处理

当检查失败时：

1. 阻止提交：不执行 `git commit`
2. 精确列出失败点：命令、文件、日志、复现方式、风险说明至少要有一项
3. 先修复再重跑：只修报告里提到的问题不够，要重新验证相关风险面
4. 循环直到通过：全部关键验证通过后才能进入下一步 review 或提交

以下情况一律视为 **未通过**：

- 任务目标无法被直接验证
- 关键测试失败或未运行
- 有高风险改动但没有对应证据
- 存在已知架构/安全问题却选择带病提交
- 交付类任务未满足仓库要求的流程闭环，却宣称任务已完成
- 报告里写了"理论上应该没问题"，但没有执行结果支撑

## 与项目 Agent 指令集成

如果项目使用 Agent 指令文件（如 `CLAUDE.md`、`AGENTS.md`），建议在其中的代码提交流程部分加入自审判断逻辑：

```markdown
**Step 2：开发代码 + 本地自审**
- 遵循本文档定义的代码规范
- 完成后判断是否需要自审：
  - **会影响行为、交付物或验证链路的改动**：运行 self-review skill
  - **纯说明性文档改动**：可跳过 self-review
- 自审通过后再提交
```
