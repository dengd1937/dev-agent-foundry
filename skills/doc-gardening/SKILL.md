# Doc Gardening

> 本 skill 把文档视为工程资产，而不是说明性附件。目标是发现并处理"文档不再真实反映仓库"的问题。

## 角色定位

- `design-review`：审方案、架构和设计决策
- `code-review-expert`：审代码变更与回归风险
- `doc-gardening`：审文档与仓库现实是否一致，并在证据充分时直接修复

它关注的不是文案润色，而是以下漂移：

- 文档描述的流程与当前代码实现不一致
- 文档引用的路径、命令、配置项、测试入口已经失效
- Agent 指令文件、文档索引、模块文档之间导航断裂
- 测试规范、交付规范与实际仓库结构不一致

## 适用场景

- 改完代码后，检查文档是否漏改
- 改完文档后，检查是否仍与代码一致
- 针对某个模块做文档治理
- 在提交前做一次文档一致性检查

## 工作模式

仅支持以下两种模式：

### 1. `review`

只审查，不改文件。

输出内容：

- `Summary`
- `Findings`
- `Repair Plan`
- `Open Questions`

### 2. `review-and-fix`

先审查，再修复证据充分且低歧义的问题。

输出内容：

- `Summary`
- `Fixed`
- `Remaining Findings`
- `Follow-up Plan`

## 审查范围

优先根据用户请求缩小范围，避免默认全仓重扫。

常见范围：

- 全仓文档
- 某个模块，如 `workflow`、`webhook`、`testing`
- 最近改动涉及的文档与代码
- 某类规范，如测试、配置、交付流程

若用户未指定范围，默认从以下入口开始：

1. 项目级 Agent 指令文件（如 `CLAUDE.md`、`AGENTS.md`）
2. 文档索引文件（如 `docs/index.md`、`docs/README.md`）
3. 与任务直接相关的模块文档
4. 对应实现代码与测试

## 核心检查项

### 1. 导航漂移

- Agent 指令文件与文档索引是否仍指向真实入口
- 文档索引是否存在孤儿文档、失效链接、遗漏入口
- 新增文档是否已进入导航

### 2. 实现漂移

- 文档描述的入口、职责、链路是否仍对应当前代码
- 文档引用的文件、函数、命令、配置项是否仍存在
- 文档中的默认行为、异常路径、降级逻辑是否仍成立

### 3. 测试漂移

- 测试规范文档是否与测试目录当前结构一致
- 文档宣称的测试分层、E2E 口径是否与实际测试相符
- 关键链路文档是否声称"已覆盖"，但仓库中没有对应测试

### 4. 配置漂移

- 文档中的配置项、环境变量、默认值是否与配置源码一致
- 是否出现文档要求通过统一配置访问，但代码/文档示例不一致

### 5. 流程与交付规则漂移

- Agent 指令文件、文档目录、skill 文档之间的规则是否冲突
- PR、review、测试、交付流程文档是否仍对应当前做法

## 工作流程

### 第一步：建立事实源

先把仓库文档和代码作为唯一事实源，不依赖聊天记录或口头约定。

至少检查：

- 用户指定范围内的文档
- 对应代码入口
- 对应测试或验证脚本

优先使用：

- `rg` / `rg --files` 搜索文档和代码引用
- 文档导航和模块设计文档定位入口
- 关键实现文件确认当前行为

### 第二步：形成对照

对每个候选漂移项，至少建立一组"文档陈述 vs 代码/测试事实"的对应关系。

没有证据不要报 finding。

以下情况可直接判为高优先级：

- 会误导开发、测试、排障或交付
- 文档路径、命令、配置项已失效
- 文档与代码对同一行为给出互相矛盾的结论

### 第三步：输出 findings

每条 finding 必须包含：

- 严重级别
- 文档位置
- 对应代码/测试证据
- 为什么算漂移
- 建议修复方向

严重级别：

- `P1`：会误导开发、测试、交付，或导致错误操作
- `P2`：信息过时但不阻塞主流程
- `P3`：轻微命名、入口、措辞不一致

### 第四步：执行修复

仅在 `review-and-fix` 模式下执行。

可直接修复：

- 链接错误
- 索引漏挂
- 文件路径变更
- 证据充分的过时命令
- 与代码一一对应、无歧义的描述失真

不要直接修复：

- 架构意图不明确
- 多个文档互相冲突且无法判断谁是新事实源
- 涉及产品决策、设计取舍或尚未完成的计划

## 输出格式

### `review`

```markdown
## Summary

- 审查范围
- 证据来源
- 总体判断

## Findings

### P1
1. [path] 标题
   - 文档陈述
   - 代码/测试证据
   - 风险
   - 建议修复

### P2
...

### P3
...

## Repair Plan

1. 先修什么
2. 再修什么
3. 哪些需要人工确认

## Open Questions

- 若无则写 `None`
```

### `review-and-fix`

```markdown
## Summary

- 审查范围
- 修复范围
- 未修部分的原因

## Fixed

1. [path] 修复内容
2. [path] 修复内容

## Remaining Findings

### P1
...

### P2
...

## Follow-up Plan

1. 还需要补什么
2. 需要谁确认
```

## 边界要求

- 默认先做最小必要检查，不要无目标扫全仓
- 默认最小修改，不做无关重写
- 发现文档与代码不一致时，优先判断"谁更接近事实"，不要机械改文档
- 若证据不足，明确说明不足并保留为 finding，不强行修复
- 若修复涉及行为性规则或流程更新，需同步更新导航入口

## 与项目文档体系的关系

- 统一入口通常是项目的 Agent 指令文件和文档索引
- 文档写作规范参照项目自身的文档指南（如有）
- 测试分层与 E2E 策略参照项目自身的测试规范（如有）

如用户请求创建修复计划而不改文件，使用 `review`。
如用户明确要求"审查并修复"，使用 `review-and-fix`。
