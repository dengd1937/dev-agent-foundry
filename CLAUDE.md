# UI Generation Rules

- If `DESIGN.md` exists in the project root, read it before starting any design work and at each major design stage. Treat its color palette, typography, component styles, layout principles, depth rules, do's and don'ts, and responsive behavior as hard constraints — not suggestions.
- If `DESIGN.md` does not exist, use existing product/design artifacts and reusable components; only fall back to conservative defaults when no project-specific visual direction exists.
- DESIGN.md is default-read-only within the workflow: never silently modify it to accommodate a design. If a design need falls outside DESIGN.md, document the design identity gap and ask whether to constrain the design or intentionally update DESIGN.md.
