# Patterns

> 本文档定义跨项目通用的实现模式基线。

## 骨架项目优先

在实现新能力前：

1. 先搜索经过验证的骨架项目、模板仓库或成熟实现
2. 优先评估是否可以复用、裁剪、包装或二次集成，而不是从零开始重写
3. 在成熟结构内迭代，而不是先发明一套全新目录和抽象

## 分层与边界

- 数据访问、外部依赖调用和业务规则应保持清晰边界
- 业务逻辑应依赖稳定接口，不应直接绑定底层存储、第三方 SDK 或传输细节
- 协议适配、模型转换和输入解析应尽量收敛在边界层，而不是散落在核心逻辑中

## Repository Pattern

当项目存在数据访问层时，优先通过一致接口封装访问能力：

- 抽象出稳定操作，例如查询、创建、更新、删除
- 具体实现负责数据库、API、文件系统等存储细节
- 业务逻辑依赖接口，而不是依赖某种底层存储实现
- 这样更利于替换数据源，也更利于测试中的 mock 或 fake 实现

## API 响应格式

对外提供 API 时，优先维持一致的响应包裹格式：

- 明确 success / status 字段
- data 负载与 error 信息保持稳定约定
- 分页、游标或统计信息应进入 metadata，而不是散落在顶层

## 完成前检查

- [ ] 新功能是否先调研过成熟骨架项目、模板或现成实现
- [ ] 业务逻辑是否仍然依赖稳定接口，而不是直接耦合底层细节
- [ ] 输入解析、协议适配和模型转换是否收敛在边界层
- [ ] 对外 API 是否保持了稳定、一致的响应格式
