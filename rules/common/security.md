# Security

> 本文档定义跨项目通用的安全基线。

## 提交前安全检查

在任何实现进入提交阶段前，至少确认：

- [ ] 没有硬编码 secret、token、密码或密钥
- [ ] 用户输入与外部输入都已经校验
- [ ] 数据库操作使用参数化查询或等价防注入方式
- [ ] 输出到 HTML 或富文本内容前，已确认 XSS 风险得到处理
- [ ] 认证、鉴权和权限边界没有被绕开
- [ ] 对外接口具备必要的限流、配额或滥用防护
- [ ] 错误信息不会泄露敏感数据、内部路径或凭据

## Secret 管理

- 禁止在源码、测试样例、脚本或文档中硬编码真实 secret
- secret 应通过环境变量、secret manager 或项目约定的统一配置入口注入
- 启动时应尽早校验关键 secret 是否存在，而不是运行到深层逻辑才失败
- 一旦怀疑 secret 暴露，应先轮换，再继续后续开发动作

## 边界防护

- 所有来自用户、第三方 API、文件、Webhook、消息队列的输入都应视为不可信
- 认证与鉴权应在明确边界执行，而不是依赖调用方“应该已经做过”
- 不要把敏感字段完整写入日志、报错、追踪或审计记录

## 安全问题响应

发现安全问题时：

1. 先停止继续推进交付
2. 先修复高风险问题，再继续其他开发
3. 若存在已暴露 secret，立即轮换
4. 回看相邻代码路径，确认是否存在同类问题
