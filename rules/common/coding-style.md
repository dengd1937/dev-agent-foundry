# Coding Style

> 本文档定义跨项目通用的编码风格基线。

## 不可变性

_这是默认优先规则。_

- 优先创建新对象，而不是在原对象上原地修改
- 需要更新结构化数据时，优先返回带变更的新副本
- 仅在性能、底层接口或现有框架明确要求原地修改时，才允许例外

这样做的目的：

- 减少隐式副作用
- 降低调试成本
- 让并发、缓存和测试更安全

## 文件组织

- 优先维护高内聚、低耦合的小文件
- 单个文件通常控制在 `200-400` 行；若明显超过 `800` 行，应主动考虑拆分
- 大文件中的工具函数、适配逻辑和重复逻辑，应尽早提取
- 目录和模块优先按功能域组织，而不是仅按技术类型堆叠

## 错误处理

- 错误必须被显式处理，不要静默吞错
- 面向用户的边界应返回清晰、可理解的错误信息
- 服务端、任务执行器或后台逻辑应保留足够上下文，便于定位根因
- 捕获异常后若继续抛出，应尽量保留原始异常链

## 输入校验

- 所有系统边界输入都应先校验，再进入核心逻辑
- 用户输入、API 返回、文件内容、消息队列负载、环境变量都属于不可信输入
- 有 schema、模型或类型系统时，优先使用结构化校验，而不是散落的 `if` 判断
- 发现非法输入时应尽早失败，并给出明确错误

## 命名与可读性

- 名称应直接表达意图，避免使用模糊缩写
- 函数应尽量短小、单一职责；明显超过 `50` 行时应检查是否可拆分
- 避免超过 `4` 层的深层嵌套；优先使用提前返回、辅助函数或状态收敛降低复杂度
- 不要把关键业务含义藏在魔法数字、硬编码字符串或隐式约定中

## 完成前检查

在标记完成前，至少确认：

- [ ] 代码可读，命名清晰
- [ ] 函数保持短小，职责单一
- [ ] 文件保持聚焦，没有继续无节制膨胀
- [ ] 没有不必要的深层嵌套
- [ ] 错误处理完整，没有静默吞错
- [ ] 边界输入已经校验
- [ ] 没有新增不必要的硬编码值
- [ ] 默认使用不可变更新模式，而不是随意原地修改
