# Performance

> 本文档定义跨项目通用的性能与效率约束。

## 先测量，再优化

- 默认不要凭直觉提前做复杂优化
- 发现性能问题时，先定位瓶颈，再决定优化方向
- 优化应围绕真实热点、慢路径和高成本操作展开，而不是泛化重构

## 上下文与任务规模控制

- 大规模改动、跨多文件重构和复杂交互排障，应先拆分任务，避免一次把过多上下文塞进同一步骤
- 单文件修复、独立工具函数和纯文档变更，应保持轻量处理，不要过度规划
- 复杂任务优先先产出计划，再进入实现

## I/O 与重复计算

- 优先减少不必要的数据库查询、网络请求、磁盘读取和重复序列化
- 对明显重复且结果稳定的计算，可考虑缓存、批处理或结果复用
- 不要在循环中反复执行高成本 I/O，除非已经验证这是必要行为

## 渐进式修复

- 构建失败、性能回退或复杂问题排查时，应按最小步修复并逐步验证
- 每次修改后都应重新确认问题是否缩小，而不是一次性叠加多种猜测性修复

## 完成前检查

- [ ] 是否先定位了真实瓶颈，而不是凭感觉优化
- [ ] 复杂任务是否已经拆分并形成明确计划
- [ ] 是否去掉了明显重复的 I/O、查询或计算
- [ ] 排障和优化是否按小步验证推进，而不是一次性堆叠改动
