# Testing

> 本文档定义跨项目通用的测试要求。

## 最低覆盖要求

- 默认目标覆盖率不少于 `80%`
- 覆盖率是基线，不是唯一目标；关键路径和高风险逻辑应优先保证有效测试

## 必要测试类型

以下三类测试默认都应具备：

1. **单元测试**
   覆盖函数、工具模块、纯逻辑组件
2. **集成测试**
   覆盖 API、数据库操作、模块协作和边界交互
3. **端到端测试**
   覆盖关键用户流程或关键业务链路

## TDD 工作方式

默认采用以下顺序：

1. 先写测试，进入 `RED`
2. 运行测试，确认它确实失败
3. 编写最小实现，进入 `GREEN`
4. 再次运行测试，确认它通过
5. 重构和清理，进入 `IMPROVE`
6. 最后确认覆盖率和回归情况

## 失败排查

测试失败时，优先按这个顺序检查：

1. 测试是否具备隔离性
2. mock、stub、fixture 是否正确
3. 是实现错了，还是测试预期本身错了
4. 优先修实现；只有测试本身错误时才改测试

## 完成前检查

- [ ] 覆盖率是否达到最低目标，或已有明确例外说明
- [ ] 单元、集成、端到端三类测试是否覆盖到与本次改动匹配的层级
- [ ] 是否按 `RED -> GREEN -> IMPROVE` 顺序推进，而不是先写实现再补测试
- [ ] 测试失败时，是否先检查隔离性、mock 和实现，而不是直接改断言
